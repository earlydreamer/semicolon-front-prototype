## 4) Git Flow 전략 (실 운영 기준)
### 기본 브랜치 구조
- main — 배포(운영)
- develop — 통합 개발
- feature/* — 기능 개발
- hotfix/* — 운영 긴급 수정
- release/* — 릴리즈 준비
---
### 기능 개발 브랜치 예시
- feature/auth-login
- feature/product-search
- feature/order-cart
- feature/payment-deposit
- feature/settlement-batch
**머지 기준**
- 최소 1인 코드리뷰
- 테스트 통과 필수
- 린트 자동검사
---
### 릴리즈 프로세스
1. develop → release/x.x.x 생성
2. QA / 점검 / 마이그레이션 테스트
3. release → main 반영
4. 태깅
- v1.0.0
- v1.1.0
- v1.1.1
1. main → develop 역머지
---
### 운영 핫픽스
예시
- hotfix/payment-timeout
- hotfix/order-rollback
**절차**
- main에서 hotfix 분기
- 수정 후 main + develop 동시 반영
---
### 커밋 컨벤션
- feat: 기능 추가
- fix: 버그 수정
- refactor: 리팩토링
- chore: 빌드/환경
- docs: 문서
- style: 포맷
- test: 테스트
**예시**
- feat(product): 카테고리별 정렬 기능 추가
- fix(order): 결제 실패 시 재시도 로직 수정
